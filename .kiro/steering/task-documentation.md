# Task Documentation Requirements

## Summary Documentation

At the completion of each task, you MUST save a copy of your task summary in the `docs/` folder as a separate document. This creates a permanent record of work completed and decisions made.

### Documentation Process

1. **Generate Summary**: Create your normal task completion summary as usual
2. **Save Copy**: Save an identical copy of that summary to `docs/` folder
3. **File Naming**: Use descriptive filenames with timestamps for easy reference

### File Naming Convention

Use the following format for documentation files:
```
docs/YYYY-MM-DD-task-description.md
```

Examples:
- `docs/2024-12-17-logging-system-implementation.md`
- `docs/2024-12-17-tenant-dashboard-updates.md`
- `docs/2024-12-17-aws-service-integration.md`

### Content Requirements

The documentation should include:
- **Task Overview**: Brief description of what was accomplished
- **Key Changes**: List of files modified, created, or deleted
- **Technical Decisions**: Important architectural or implementation choices made
- **Testing**: Any tests run or testing considerations
- **Next Steps**: Recommendations for follow-up work (if applicable)

### Purpose

This documentation serves multiple purposes:
- **Project History**: Maintains a chronological record of development progress
- **Knowledge Transfer**: Helps team members understand past decisions and changes
- **Debugging**: Provides context when investigating issues or regressions
- **Planning**: Informs future development and refactoring efforts

## Implementation Note

This is a mandatory step - do not consider a task complete until the summary documentation has been saved to the `docs/` folder.
